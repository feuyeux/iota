# Iota Engine 实现与设计有歧义的内容

> 日期：2026-04-25  
> 来源：docs/iota_engine_design_0424.md 审查结果

本文档记录实现与设计文档存在歧义或需要澄清的内容。

---

## 1. 记忆系统命名差异

### 设计文档
设计文档 Section 12.1 定义了五种记忆类型：
- Conversation Memory
- Working Memory (`working_memory`)
- Factual Memory (`factual_memory`)
- Episodic Memory (`episodic_memory`)
- Procedural Memory (`procedural_memory`)

### 实际实现
实际代码中 `MemoryKind` 类型定义为：
```typescript
export type MemoryKind = 'episodic' | 'procedural' | 'factual' | 'strategic';
```

### 歧义
- 设计文档中没有 `strategic` 类型
- 实现中缺少 `working_memory` 作为 `MemoryKind` 枚举值
- 设计文档明确说明 "Conversation Memory 是运行时对话上下文组件，不作为 `MemoryKind` 写入向量或关键词索引"

### 建议
需要确认：
1. `strategic` 是否应该改为 `working_memory`？
2. 还是 `strategic` 是新增的记忆类型，需要补充到设计文档？
3. Working Memory 是否应该作为 `MemoryKind` 的一个值？

## 2. Routing 策略简化

### 设计文档 Section 6.1
设计文档明确说明：
> "Backend 选择为纯入参指定，不引入自动路由计算逻辑"
> "不再维护 fallback chain、能力匹配、亲和性评分和成本/延迟策略等复杂路由逻辑"

### 实际实现
`BackendResolver` (routing/resolver.ts) 的实现需要确认是否完全符合这一简化原则。

### 需要确认
1. 当前 `BackendResolver` 是否只做简单的 backend 指定和 defaultBackend 回退？
2. 是否存在任何自动路由计算逻辑？
3. 是否存在 fallback chain 实现？

## 3. MCP 集成范围

### 设计文档 Section 9.2
设计文档说明：
> "Claude Code 为单次执行进程，无法在执行期间代理 MCP 连接，因此 `mcp: false`"
> "Gemini CLI 内置了自己的 MCP 客户端，通过自身配置直接连接 MCP Server，不经过 Iota MCP Router（Iota 视角 `mcp: false`）"
> "MCP Router 仅服务于 Codex 和 Hermes 等长驻后端"

### 实际实现
需要确认各 adapter 的 `capabilities.mcp` 字段是否正确设置：
- ClaudeCodeAdapter: 应该是 `mcp: false`
- GeminiAdapter: 应该是 `mcp: false`
- CodexAdapter: 应该是 `mcp: true`
- HermesAdapter: 应该是 `mcp: true`

### 需要验证
检查 backend adapter 实现中的 capabilities 定义是否与设计文档一致。

## 4. 审批流程的 Native Backend Approval

### 设计文档
设计文档在多处提到审批机制：
- Section 7.2: Claude Code 的 `control_request` / `control_response`
- Section 7.3: Codex 的 `CommandExecutionRequestApproval`
- Section 7.5: Hermes 的 `request_permission`
- Section 15.1: Iota 层面的 6 类审批类型

### 实际实现
Engine 中同时处理：
1. Native backend approval (通过 `approval_request` extension event)
2. Iota 层面的 approval policy (通过 `guardEvent` 和 `enforceApprovalPolicy`)

### 需要澄清
1. 两层审批的优先级和交互关系
2. 是否存在重复审批的情况？
3. Gemini 的 `--approval-mode yolo` 是否会绕过 Iota 的 path/network guard？

## 5. Subprocess 进程模型

### 设计文档 Section 2.4
设计文档说明：
> "Claude Code 和 Gemini CLI 默认每次 execution 启动一个短进程"
> "Codex、Hermes 默认由 Backend Pool 保持 warm process"

### 实际实现
需要确认：
1. ClaudeCodeAdapter 和 GeminiAdapter 是否真的是 per-execution spawn？
2. CodexAdapter 和 HermesAdapter 是否维护长驻进程？
3. 长驻进程的 `activeWarmExecution` 串行化模型是否已实现？

## 6. 事件序号分配

### 设计文档 Section 8.1
> "每个 execution 的事件从 `sequence = 1` 单调递增。序号由 Engine 分配，adapter 不直接决定最终 sequence"

### 实际实现
需要确认：
1. Adapter 返回的事件是否带有 `sequence = 0` 或不带 sequence？
2. EventStore 是否在 `append()` 时统一分配 sequence？
3. 是否存在 adapter 自己分配 sequence 的情况？

## 8. Fencing Token 实现范围

### 设计文档 Section 13.4
设计文档说明：
> "每次 `acquireLock` 通过 Redis `INCR` 递增 monotonic token"
> "`assertFencingValid()` 在每个关键写入点前调用"

### 实际实现
需要确认：
1. RedisStorage 是否实现了 `validateFencingToken` 方法？
2. 所有关键写入点是否都调用了 `assertFencingValid()`？
3. Fencing token 是否真的通过 Redis INCR 实现？

1. 用户级配置 `~/.iota/config.yaml` 是否被支持？
2. 项目级配置优先级是否正确实现？
3. 环境变量覆盖是否正确实现？

## 10. 统一大模型配置的映射

### 设计文档 Section 14.3.1
设计文档说明：
> "各 backend adapter 负责将这组统一配置映射为自身 CLI 所需的参数格式（如 `--model`、`--api-key` 等）"

### 需要验证
需要检查每个 adapter 的实现，确认：
1. 是否从环境变量读取 `IOTA_BASE_URL` / `IOTA_MODEL` / `IOTA_AUTH_TOKEN`？
2. 是否正确映射为各自 CLI 的参数格式？
3. 是否存在 adapter 自己维护独立配置的情况？

## 总结

以上歧义主要集中在：
1. **命名一致性**：MemoryKind 类型定义
2. **能力标记**：Backend capabilities 的 mcp 字段
3. **审批流程**：两层审批的交互关系
4. **进程模型**：长驻 vs per-execution 的实际实现
5. **配置映射**：统一大模型配置的实际使用

建议：
- 对照设计文档逐一验证实现细节
- 更新设计文档以反映实际实现（如果实现是正确的）
- 或修改实现以符合设计文档（如果设计是正确的）
